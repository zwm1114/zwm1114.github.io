廖雪峰Git教程笔记整理

2019-07-13

几个常用命令：

查看哪些文件修改了：git status

查看修改的内容：git diff

查看提交日志：git log

简化日志信息：git log --pretty=oneline

一、版本管理

1.把当前版本回退到上一个版本

git reset --hard HEAD^

git <filename> //查看文件内容

2.回退到上一个版本之后又想回到这个版本

找到该版本号（无需写全，只需写前几位，多写几位不然找不到的…）

git reset --hard <id>

3.当前窗口已关闭，还想回到某一个版本

git reflog //找到版本id

git reset --hard <id>

二、工作区与版本库

理解他们很重要!

commit 后，用 git diff HEAD -- <filename> 查看工作区和版本库里最新版本的区别。

1.没使用add和commit

git checkout -- <filename> //丢弃文件的修改内容

2.使用了add ，未使用 commit

git reset HEAD <filename> //相当于撤销掉了add命令

3.使用了add和commit命令

版本回退

三、删除文件

add和commit 之后想删除文件

1.确实想删除

git rm <filename>

git commit -m “XXX”

2.手滑

git checkout -- <filename> //一键还原，就是用版本库里的版本替换工作区的版本

ps:从未添加进版本库里的被删除的文件是无法恢复的!

四、远程仓库

1.初始化本地仓库

git init

2.添加远程仓库

git remote add <name> <SSH/Https>

3.把本地库的内容推送到远程库

如果远程库是空的：git push -u <name> <branchname> //第一次推送，本地分支与远程分支关联起来

非空：git push <name> <branchname> //提交

4.从远程库克隆

git clone <SSH/Https> //推荐使用SSH

五、新建分支

1.创建分支

git checkout -b <branchname>

2.查看当前分支

git branch

3.切换回某个分支

git checkout <branchname>

4.合并分支

git merge <branchname>

ps:这种合并是快速合并，另外有 --no-ff方式合并分支

git merge --no-ff -m ‘XXX’ <branchname>

5.合并完分支可以删除分支

git branch -d <branchname> //删除分支

六、解决冲突

1.当两个分支都有了各自的提交

需要手动修改不同的内容，然后add，commit

ps: git log --graph --pretty=oneline --abbrev -commit

七、分支管理策略

1.最佳实践：

1）master分支十分稳定，用来发布新版本

2）dev是用来工作的分支

3）团队成员有自己的分支，往dev上合并

2.bug分支

git stash //保护现场

git checkout <branchname> //选择在哪个分支上修复bug

git checkout -b <name> //创建分支

git checkout <branchname> //切换回原来的分支

git merge --no-ff -m ‘XXX’ <name> //采用--no-ff方式合并分支

git branch -d <name> //合并完，删除分支

git checkout <branchname> //切换到原来工作的分支上

git stash list //查看现场

//恢复现场

1）

git stash apply //恢复

git stash drop //删除stash内容

2）

git stash pop //恢复的同时把stash内容也删除了

ps:可以多次恢复，然后统一drop

//恢复指定内容

git stash //查看

git stash apply stash @{0} //恢复指定内容

//丢弃一个没有合并过的分支

git branch -D <branchname>

八、多人协作

git remote //查看远程看的信息

git remote -v //显示更详细的信息

git push origin <branchname> //推送分支

git checkout -b <localbranchname> <remotename>/<remotebranchname> //创建远程的dev分支到本地

？两人提交的代码有冲突怎么办

git pull 会失败

git branch --set-upstream-to = <name>/<branchname> <branchname> //指定本地branch与远程branch连接

git pull //再pull

九、Rebase

先pull别人的然后再git rebase。

作用：好看！把本地未push的分叉提交历史整理成直线，使查看历史提交变化时更容易。

十、标签管理

1.创建标签

git checkout <branchname> //先切换到该分支上

git tag <tagname> //打标签

git tag //查看所有标签

ps:默认标签是打在最新提交的commit上

若想在之前的commit上打标签，先找到历史提交的commit id

git tag <tagname> <id>

git show <tagname> //查看标签信息

git tag -a <tagname> -m ‘XXX’ <id> //创建带有说明的标签

ps:标签总是和某个commit挂钩的，如果这个commit出现在两个不同的分支，那么这两个分支都可以看到这个标签

2.操作标签

git tag -d <tagname> //删除标签

git push <name> <tagname> //推送某个标签到远程

git push <name> --tags //推送全部未推送到远程的本地标签

若标签已推送到远程

git tag -d <tagname> //先本地删除

git push <name> :refs/tags/<tagname> //从远程删除

其他：git remote rm <name> //删除已有的github远程库

十一、自定义git

1.忽略特殊文件

没懂…

2.配置别名

git config --global alias.<replacename> <name>

例如：git config --global alias.lg “log --graph --pretty=oneline --abbrev-commit”

​ 输入git lg 即可查看分支合并情况

配置信息在隐藏目录 .git/config 中

ps:global参数是全局参数，也就是这些命令在这台电脑的所有git仓库下都有用

十二、个人遇到的错误及解决办法

refusing to merge unrelated histories* 本地仓库和远程不是同一个仓库，无法合并两个独立的仓库历史

解决办法：git pull <name> <branchname> --allow-unrelated-histories

​ git push <name> <localbranch>:<remotebranch>

0.1+0.2=?

2019-07-10

为什么会出现以下问题？

其实这个问题和1/3为什么是0.3333333333…是一个道理的。

首先需要明白浮点数在计算机中是如何存储的。

float型数据：长度是4个字节，第1位是符号位，中间8位是指数位，剩下23位是尾数位。

double型数据：长度是8个字节，第一位是符号位，中间11位是指数位，剩下52位是尾数位。

举个栗子：2.15在计算机中的存储

首先需要将2.15转换为二进制10.001001100110011001100110011001100110011001100110011。整数部分转化为二进制是10，小数部分依次乘2将得到的整数部分正序排列。

用科学计数法表示为 1.000100…×2^1

这里假设是float类型，double类型也是一个方法的。

1）符号位：1表示负值，0表示正值。

2）指数位：指数的真实值 = 指数部分的整数 + 偏差值。对于32位浮点数，偏差值为127，对于64位浮点数，偏差值为1023。

3）尾数位：去掉小数点的那部分。

这里是不是发现了一个问题，如果是无限循环小数的话，4位或者是8位的存储空间是根本存不下的，这就导致了精度缺失。

0.1和0.2转换为二进制都是无限循环小数。JavaScript的Number类型数据遵循是IEEE754规范，采用双精度存储，其他家使用相同数值格式的语言也存在这个问题，所以在Java中也可以看到。

0.1+0.2 ！= 0.3是因为计算机认为0.30000000000000004不够近似于0.3，哈哈…

理解React生命周期

2019-07-04

生命周期图示参考：

http://projects.wojtekmaj.pl/react-lifecycle-methods-diagram/

组件的生命周期有Mounting,Updating,Unmounting三个阶段

Mounting是将组件挂载到浏览器的DOM中

​ constructor:初始化state、绑定成员函数的this环境
​ getDerivedStateFromProps:在调用render()之前调用，并在每次渲染时调用（不常用）
​ componentDidMount:首次渲染后调用，所有的AJAX请求，DOM或状态更新、设置监听器都应该在此处发生

Updating是组件更新

​ shouldComponentUpdate:确定组件是否更新，允许组件接收新属性时阻止重新渲染

ps:当return false 时，阻止组件重新渲染，如果不在组件内部中写这个函数，默认是return true,但是如果写了这个函数，没有写返回值的话，会出现警告，并且阻止组件渲染。

​ getSnapshotBeforeUpdate:在最新的渲染输出提交给DOM前会立即调用，从DOM捕获信息很有用(不常用)
​ componentDidUpdate:更新DOM,以响应props和state的更改

Unmounting是将组件从浏览器的DOM中卸载

​ componentWillUnmount:当一个组件被从DOM中移除时，该方法被调用，取消网络请求或者移除与该组件相关的监听程序等都应该在这里进行
